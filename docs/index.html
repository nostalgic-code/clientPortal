System Architecture

The portal uses a decoupled frontend-backend architecture with Next.js proxying API calls to a Flask backend. All data is stored in Supabase (hosted PostgreSQL) with Row-Level Security policies.

Authentication System

Authentication is handled by Supabase Auth with ES256 JWT tokens. The system supports two roles: admin (agency staff) and client (invited clients).

Auth Flow — Admin Registration

1Registeremail + password
→
2Supabasesign_up()
→
3Create Orgorganizations table
→
4Link Userorg_users + profile
→
5Return JWTaccess + refresh

Auth Flow — Client Invitation

1Admininvites client
→
2Create Useradmin.auth.create_user
→
3Set Metadatarole: 'client'
→
4Link Clientclients.user_id = id
→
5Add to Orgrole: 'member'

Token Validation

supabase_auth_required decorator
@wraps(f)
def decorated(*args, **kwargs):
    token = request.headers.get('Authorization', '').replace('Bearer ', '')
    if not token:
        return jsonify({'error': 'Missing token'}), 401
    
    # Validate against Supabase Auth API
    headers = {
        'Authorization': f'Bearer {token}',
        'apikey': SUPABASE_KEY
    }
    resp = requests.get(f'{SUPABASE_URL}/auth/v1/user', headers=headers)
    
    if resp.status_code != 200:
        return jsonify({'error': 'Invalid or expired token'}), 401
    
    request.supabase_user = SimpleNamespace(**resp.json())
    return f(*args, **kwargs)

Frontend Auth State

auth-context.tsx — Token Management
// Auto-refresh on 401 response
const fetchApi = async (url, options) => {
  let response = await fetch(url, {
    ...options,
    headers: { 'Authorization': `Bearer ${token}`, ...options?.headers }
  });
  
  if (response.status === 401) {
    // Refresh token via Supabase
    const { data } = await supabase.auth.refreshSession();
    if (data?.session) {
      localStorage.setItem('token', data.session.access_token);
      // Retry with new token
      response = await fetch(url, { ...options, headers: { 'Authorization': `Bearer ${data.session.access_token}` }});
    }
  }
  return response;
};

All API endpoints return consistent JSON error responses with appropriate HTTP status codes.

  "error": "Human-readable error message"
}

HTTP Status Code Reference

CodeMeaningCommon Scenarios
200SuccessGET, PUT, POST (actions like send/sign/pay)
201CreatedPOST create operations (clients, proposals, invoices)
400Bad RequestMissing required fields, invalid status transition
401UnauthorizedMissing/expired JWT token
403ForbiddenClient accessing admin routes, wrong organization
404Not FoundResource doesn't exist or not in user's organization
409ConflictClient already invited, duplicate email
500Server ErrorSupabase errors, unexpected exceptions

Security Boundaries

Organization Isolation

All queries filter by organization_id. Admin A cannot see Admin B's data.

Role-Based Access

Admin: full CRUD. Client: read own data only. Enforced at blueprint level.

Row-Level Security

Supabase RLS policies provide defense-in-depth. Backend uses service role key.

Token Validation

Every protected endpoint validates JWT against Supabase Auth API. No local JWT secrets.
